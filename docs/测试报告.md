## 三、P0/P1 详情与修复方案

修复代码全部在 **`/home/qi/type/fixes/`** 目录，按编号对应。以下为要点。

### B1 剪贴板恢复机制不可靠（fixes/01）

**根因**：`OleGetClipboard` 返回的 `IDataObject` 是包装当前剪贴板的**活代理**，不是快照（对方进程数据、或按需读取剪贴板的包装器）。随后 `copy_text` 里的 `EmptyClipboard` 已经把原始内容销毁，`OleSetClipboard(original)` 恢复的要么是已失效的代理（失败，报 CLIPBOARD_RESTORE_FAILED），要么是我们自己刚写入的文本（静默丢失用户剪贴板）。README 的核心卖点"保存所有剪贴板格式……恢复原剪贴板"在最常见场景（用户从记事本/浏览器复制的普通内容）下无法成立。

**失败场景**：用户复制了一段重要文字 → 按热键说话 → Rain 粘贴识别结果并"恢复"剪贴板 → 用户 Ctrl+V 粘贴旧内容 → 粘出来的是识别文本或空内容，原剪贴板内容丢失。

**修复**（fixes/01 提供完整可替换代码）：在**同一个 OpenClipboard 会话**内先 `EnumClipboardFormats` 逐格式 `GetClipboardData` + `GlobalSize/GlobalAlloc/memcpy` 深拷贝（CF_ENHMETAFILE 用 `CopyEnhMetaFile`，跳过 CF_BITMAP/CF_PALETTE/CF_METAFILEPICT 等 GDI 句柄格式和 0x80-0xFF 所有者渲染格式——位图内容由 CF_DIB 副本覆盖，系统恢复后会自动重新合成），再清空并写入识别文本；恢复时逐格式 `SetClipboardData` 移交所有权，未移交的句柄由 Drop 释放。附带收益：`windows` crate 依赖可整体移除（现在只有 OLE 用到它）。

### B2 粘贴/恢复竞态（fixes/01 一并修复）

**根因**：`send_paste()` 发出合成 Ctrl+V 后只 `sleep(80ms)` 就恢复剪贴板。目标应用处理按键→执行粘贴→读剪贴板是异步的；Electron 应用、远程桌面、高负载时超过 80ms 很常见。

**失败场景**：识别完成 → 写入剪贴板 → Ctrl+V → 80ms 后恢复旧剪贴板 → 目标应用 120ms 时才读剪贴板 → **文档里出现的是用户的旧剪贴板内容**，且无任何错误提示。

**修复**：等待时间放宽到 300ms（fixes/01 已内置）；如需更稳可做成配置项。顺带修复：发送 Ctrl+V 前用 `GetAsyncKeyState` 检测并补发仍按住的 Shift/Alt/Win 的 KEYUP——按住说话模式下用户常在识别完成时仍按着 Ctrl+Shift，目标应用会收到 Ctrl+Shift+V（不少应用是"无格式粘贴"或其他语义）。

### B3 HiDPI 下浮条与取消按钮错位（fixes/02）

**根因**：`work_area()` 返回 `GetMonitorInfoW` 的**物理像素**，`set_position(PhysicalPosition)` 也是物理像素，但 `440 / 92 / 350 / 17` 全是逻辑设计尺寸。数值验证（150% 缩放）：浮条物理宽 660 却按 440 居中（右偏 110px）；`y = top+height-92`，浮条物理高 108 → 底边超出工作区 16px（被任务栏遮住）；取消按钮窗口摆在浮条物理 x+350 处 ≈ 逻辑 233px，正好悬在浮条中间——而浮条自带的那个"Esc 取消"按钮因 `set_ignore_cursor_events(true)` 根本点不到（它只是装饰，真正可点击的是独立的 overlay-cancel 窗口）。100% 缩放时两者恰好像素级重合，说明只按 100% 设计过。

**修复**：用 `app.monitor_from_point(工作区中心)` 拿目标显示器 `scale_factor()`，所有逻辑常量乘以缩放系数后再参与物理坐标运算（fixes/02 提供完整代码块）。

### B4 单键全局热键劫持（fixes/04）

`hotkeyFromEvent` 允许返回不含修饰键的单键，`Config::validate` 只拦空串和 Escape，global-hotkey 0.8 接受 `"A"`。注册后字母 A 在全系统被吞。修复：JS 端要求至少一个修饰键（无修饰键仅放行 F13-F24），config.rs `validate` 加同规则双保险（防手改 config.json 绕过）。

### B5 auto 模式无 GPU→CPU 回退（fixes/05）

`fail_task` 的回退判断改为 `matches!(config.device_preference.as_str(), "cuda" | "auto")` 一行即可。默认配置 `auto` + NVIDIA 卡上 worker 会选 cuda:0，OOM 后现在只会反复失败。

### B6 热键路径上的 nvidia-smi（fixes/05）

`start_recording`（在主线程的快捷键回调里执行）→ `configure_worker_runtime` → `status()` → `detect_nvidia()` 同步等待 `nvidia-smi` 退出。修复：探测结果用 `OnceLock` 进程级缓存（显卡不会中途出现），`refresh_runtime_status` 可选强制重探；同时建议把 `InputTarget::capture()` 提到 `start_recording` 最前（目前排在一串磁盘 IO 之后，期间用户可能已切走焦点）。

### B7 Worker 取消协议地雷（fixes/03，已实测复现）

实测记录（真实 `rain_worker.py` 进程，假 funasr）：

```
cancel ack      -> {'event': 'cancelled', 'request_id': 'T2'}
started         -> {'event': 'transcription_started', 'request_id': 'T2'}
terminal for T2 -> TIMEOUT（无任何终结事件）
（worker 本身仍存活，后续 get_status 正常应答）
```

当前 Rust 从不发送 `cancel`，所以未爆雷；但这意味着：(a) 用户取消后推理照跑，Worker 互斥锁被占用，**紧接着的下一次录音在"加载模型"阶段排队等被取消的推理跑完**（长音频 CPU 推理可达几十秒）；(b) 未来谁接上 cancel 命令，Rust `read_until` 将等满 600 秒、误杀 worker 并计入崩溃熔断。修复：worker 在不提交时发 `transcription_cancelled`（终结事件永远存在），Rust `read_until` 接受该事件并映射为已有的 `CANCELLED` 丢弃分支（fixes/03 含两侧代码）。

### B8 键盘自动重复（fixes/04，需 Windows 实测 1 分钟）

Windows 按住 `RegisterHotKey` 热键会按键盘重复速率反复投递 WM_HOTKEY；global-hotkey 0.8 是否滤重我无法在本机确认。若不滤：按住说话时 BUSY ephemeral 每秒弹数十次、浮条在"正在录音/正在识别"间闪烁（并因 B25 乱跳位置）；开关模式则开始/结束抖动。防御修复对两种结果都无副作用：push_to_talk 的 Pressed 分支里 `if recording { return; }`；toggle 建议改在 Released 上切换或加 300ms 去抖。

## 六、值得肯定的设计（保持现状）

- 状态机 + request_id 贯穿：迟到/取消的识别结果不会误提交（`accepts_transcription` 有测试覆盖，我的进程级测试也间接验证了 ID 过滤）。
- 路径安全一致性好：`path_from_manifest`、`relative_path`、`enclosed_name`、`safe_remove_dir` 双 canonicalize 前缀校验，删除永远出不了受管目录。
- 下载链路完整：断点续传（Range + 206 校验）、逐文件 SHA-256、`.incomplete` 暂存 + 原子 rename、磁盘空间预检。
- Worker 生命周期：Job Object kill-on-close 兜底、优雅 shutdown + 500ms 强杀、连续崩溃熔断且用户可通过"检查 Worker"显式解锁。
- 隐私边界执行到位：诊断事件清洗（`sanitize_event`）、导出配置脱敏、崩溃报告默认关闭且内容极小。
- 剪贴板恢复前用 `GetClipboardSequenceNumber` 防止覆盖用户新复制的内容——这个决策函数本身是对的（问题只在快照机制 B1 与时序 B2）。
